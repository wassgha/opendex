# OpenDex — agent notes

OpenDex is an **Electron** desktop app (electron-vite + React + Tailwind v4). It is a voice-first agentic harness, generalized from a Next.js demo.

## Process model (important)
- **Main process** (`src/main/`) — Node. Owns the agent loop (`agent/chat.ts`, Vercel AI SDK `streamText`), TTS (`tts/elevenlabs.ts`), API keys, and all IPC handlers (`index.ts`). **Secrets never reach the renderer.**
- **Preload** (`src/preload/`) — `contextBridge` exposes a minimal typed `window.opendex` API (`chat`, `synthesize`). Ambient global in `opendex.d.ts`.
- **Renderer** (`src/renderer/src/`) — React UI + the voice state machine (`lib/dex/use-dex.ts`, hook `useDex`). Talks to main only through `window.opendex`. `@/` aliases `src/renderer/src`. (Note: no `Jarvis` identifiers in code — "Jarvis" is only the name of one theme.)

## Conventions
- Add a new IPC channel: declare it in `src/main/ipc/channels.ts`, handle it in `src/main/index.ts`, expose it in `src/preload/index.ts`, type it in `opendex.d.ts`.
- Main-process runtime deps go in `dependencies` (electron-vite externalizes them; electron-builder ships them). Renderer-only libs (react, etc.) can be `devDependencies` — Vite bundles them.
- Typecheck with `pnpm typecheck`. Smoke-test the agent without Electron: `pnpm smoke:chat [briefing]`.
- Build: `pnpm build` (→ `out/`); run built app: `pnpm start`; package: `pnpm dist`.
- **macOS signing — `build/entitlements.mac.plist` must contain NO XML comments.** Apple's entitlements parser (`AMFIUnserializeXML`) is stricter than a normal plist parser and rejects `<!-- … -->`, failing codesign with `AMFIUnserializeXML: syntax error` (the reported line is approximate / the next element). Use canonical `<true/>` (no space) and keep rationale out of the file. Both `entitlements` and `entitlementsInherit` point at this one file. Entitlement rationale: `allow-jit` + `allow-unsigned-executable-memory` for the WASM voice engines (Vosk/ORT) under hardened runtime; `disable-library-validation` to load the separately-signed nut-js `libnut`; `device.audio-input` mic; `network.client/server`, `files.user-selected.read-write` for the agent/TTS/STT and the Open skill.

## Config (Phase 2)
- `src/main/config/schema.ts` — `OpenDexConfig` shape + `DEFAULT_CONFIG` + `mergeConfig`. Shared types only with the renderer (imported type-only across the process boundary).
- `src/main/config/store.ts` — hand-rolled store: `config.json` (prefs) + `secrets.json` (keys encrypted via `safeStorage`) in `userData`. `applyToEnv()` pushes config + decrypted secrets into `process.env` so agent/TTS read them as before; dev `.env` is a fallback for unset secrets. **Secret values never leave main** — IPC returns only presence booleans (`PublicConfig`).
- Greeting/persona/model are resolved per-turn in the chat handler via `buildSystemPrompt({config, briefing})` (`agent/system-prompt.ts`); greeting modes: `example` (CoreViz) | `custom` | `none`.
- Renderer: `lib/use-config.ts` loads/updates config; `App.tsx` gates on `onboarding.completed` → `OnboardingWizard` else `MainExperience`. TTS engine chosen via `lib/dex/speech-engine.ts` (ElevenLabs vs `SystemSpeechEngine`).

## LLM providers (pluggable)
- `config.llm = { provider, model }`. `provider`: `apple` (free on-device, macOS/Apple Silicon, native `@meridius-labs/apple-on-device-ai`) | `openai` | `anthropic` (both BYOK) | `gateway` (Vercel AI Gateway — one key, any provider; bare model id) | `opendex` (hosted subscription — **reserved, not implemented**: `auth: "account"`, renders disabled "coming soon").
- `src/main/config/llm-providers.ts` — pure, dependency-free metadata (`LLM_PROVIDERS`: label, `kind` local/byok/managed, `auth` none/key/account, `secretName`, curated `models`, `keyUrl`, `platforms`, `comingSoon`, `supportsTools`). Imported as a **value** by both main (resolver) and renderer (picker). (Skill metadata is similarly shared, sourced once per skill from `src/skills/<name>/meta.ts`.)
- `src/main/agent/llm/resolve-model.ts` — `resolveModel(config)` maps provider→AI SDK model (or bare string for gateway); throws a user-facing reason for unset keys / unavailable Apple / the not-built subscription, surfaced as a spoken apology by the chat handler. `chatStart` calls it per-turn (`index.ts`). Apple is **dynamic-imported** (native Swift/Rust, darwin-only) so non-mac builds never load it; `checkAppleAvailability()` (via `llm:apple-availability` IPC) gates the picker.
- New secret `ANTHROPIC_API_KEY`. `DEFAULT_CONFIG.llm.provider = "gateway"` so pre-provider configs (only `llm.model`) keep working after merge.
- Renderer: `components/llm/provider-picker.tsx` (`ProviderPicker` + `useAppleAvailability` + `isProviderReady`/`defaultModelFor`) — shared by onboarding (no default; "Continue" gated until a provider is chosen + usable) and Settings → Language model. Apple shows disabled-with-reason on Macs, hidden entirely off-Mac.

## Themes (Phase 3) — full-interface
- **A theme renders the entire main experience** (visualization + status + transcript + controls), not just the visualizer. Only the settings gear and the audio-unlock overlay live outside the theme, in `App.tsx`. (Letting developers author themes in React is the long-term goal.)
- `components/themes/` — `types.ts` (`DexThemeProps`: full state + `getAmplitude`; `DexThemeDef`: `Component` + `Preview` + `order`), `registry.ts` (`getDexTheme` by `appearance.theme`; **auto-discovers** `./*/index.tsx` via `import.meta.glob`), `theme-picker.tsx` (data-driven from each theme's `Preview`). Shared chrome lives in `themes/shared/`: `minimal-shell.tsx` (solid bg, header, push-to-talk; `hideTranscript` opt; otherwise a borderless bottom transcript overlay via `overlay-transcript.tsx` that fades older lines + renders tool cards inline), `theme-top-bar.tsx`, `text-composer.tsx`, `use-amplitude.ts`. Each **theme is its own folder** with an `index.tsx` that default-exports a `DexThemeDef`: `jarvis/` (cyan Stark HUD — `jarvis-reactor.tsx` arc reactor + `hud-widgets.tsx` satellites + amplitude-reactive `HudWaveform`), `editorial/`, `dot/` (amplitude dot + overlay transcript), `cursor/` (a single centered line **typed in by the caret** via a `useTypewriter` hook; `hideTranscript`).
- `App.tsx` picks `getDexTheme(cfg.appearance.theme).Component` and renders it with all of `useDex`'s state.
- Amplitude: `lib/dex/audio-meter.ts` meters the **mic** (real listening loudness, never connected to destination). Speaking/thinking use a synthetic envelope in `use-dex.ts` (no audio routing → no autoplay/silence risk). `useDex().getAmplitude()` returns the status-appropriate 0..1 level; themes sample it via `components/themes/shared/use-amplitude.ts` (rAF → direct DOM writes, no React re-renders).
- To add a theme: create `themes/<id>/index.tsx` that default-exports a `DexThemeDef` (Component + Preview). Discovery, picker, and config wiring are automatic — see CONTRIBUTING.md.

## Wake/STT (Phase 4a) — pluggable voice input
- Config `voiceInput: { wakeMode, sttProvider }`. Wake: `manual` (push-to-talk — orb click or ⌘⇧Space global hotkey), `vosk` (free offline WASM keyword spotting, see 4b), `webspeech` (continuous scan). STT: `webspeech` (single-shot) or `openai` (cloud Whisper). All pure JS/WASM — no native modules.
- Web Speech + the keyless `manual` path stay inline in `use-dex.ts` (tightly coupled to the state machine). Vosk wake and `openai` STT are encapsulated in `lib/dex/engines/` (`vosk-wake.ts`, `cloud-stt.ts`, `wav.ts`, `types.ts`); `startMode` branches on config.
- Audio: wake + cloud capture use `@picovoice/web-voice-processor` (WVP — 16kHz Int16 frames) in the renderer. Cloud STT endpoints on silence (energy/RMS), encodes WAV, and sends bytes to main → `src/main/stt/` (OpenAI). **OpenAI key stays main-only.** (WVP is just the mic frame pump — no Picovoice account/key involved.)
- Manual mode: `useDex().pushToTalk()` + `canPushToTalk`; themes make the visualization tap-to-talk. Global hotkey registered in main (`globalShortcut`) → `push-to-talk` IPC event → `onPushToTalk` (preload) → `pushToTalk()`.
- **Barge-in (interrupt by voice while the assistant speaks)** is **wake-word triggered and always on** (`startBargeMonitor` in `use-dex.ts`): when TTS starts (`bargeOnSpeakingRef`), the configured wake engine (Vosk offline by default; Web Speech for `webspeech` mode; none for `manual` — use Stop / hotkey) runs *during playback*; hearing the wake word stops the reply (keeps the partial for context) and drops into `command` listening. It only fires on the wake word, not arbitrary speech, so it can't echo-loop on the assistant's own audio — which is why it needs no AEC and isn't a setting (the old open-mic full-sentence Web Speech monitor was the opt-in, echo-prone version this replaced). `stopBargeMonitor` is idempotent and called from every transition out of speaking (incl. `startMode`).

## Free offline engines (Phase 4b)
- STT providers now also include **`whisper-local`** (transformers.js Whisper, WASM/WebGPU — `engines/whisper-stt.ts`) and **`vosk-local`** (vosk-browser WASM — `engines/vosk-stt.ts`); wake modes include **`vosk`** (free, no-signup, hands-free keyword spotting — `engines/vosk-wake.ts`). All free, offline, no key (one-time model download, cached). `engines/frame-capture.ts` is the shared WVP capture + endpointing helper; `engines/vosk-model.ts` caches the Vosk model.
- All heavy engines are **dynamic-imported** → code-split: main bundle ~900KB; ORT WASM (~23MB), Vosk (~5.8MB), Whisper glue (~1.2MB) load only when their mode is selected, and are bundled locally (offline-capable). Local STT model is preloaded when entering wake mode so the first capture isn't blocked; `useDex().loadingModel` drives a global download banner in `App.tsx`.
- **CSP note:** Vosk's emscripten WASM requires `'unsafe-eval'` in `script-src` (ORT only needs `'wasm-unsafe-eval'`). This is set in `src/renderer/index.html` as a deliberate tradeoff; removing the Vosk engine would let us tighten it back.

## Realtime voice mode (speech-to-speech)
- `config.voice.mode`: `pipeline` (everything above/below — wake→STT→LLM→TTS) | `realtime` (one speech-to-speech model over a WebSocket; `config.realtime = { provider, model, voice, idleDisconnectSec }`, models via the Vercel AI Gateway: `openai/gpt-realtime-2`, `xai/grok-voice-think-fast-1.0`). Metadata in `src/main/config/realtime-models.ts` (pure, shared main+renderer, like `llm-providers.ts`). Uses `@ai-sdk/gateway@canary` — pinned exact (no `^`); it has no `ai` peer dep and coexists with `ai` v6.
- **The WebSocket lives in MAIN** (`src/main/agent/realtime/session-host.ts`): gateway v0 mints NO ephemeral client secret — `getToken` returns the raw `AI_GATEWAY_API_KEY`, and the SDK hard-throws in browsers — so the socket can never live in the renderer. The gateway wire protocol IS the normalized AI SDK realtime event format (the model object's codec methods are pass-throughs); session-host still calls them so a future direct-OpenAI `RealtimeModelV4` is a drop-in. Audio relays over IPC as 24kHz mono PCM16 ArrayBuffers (~2KB/40ms frames): renderer→main `realtime:client`, main→renderer `realtime:event:<sessionId>` (contract in `channels.ts`: `RealtimeClientMessage` / `RealtimeServerNotice`).
- **Renderer** (`src/renderer/src/lib/dex/realtime/`): `realtime-session.ts` (`RealtimeVoiceSession` — IPC bridge + idle-disconnect window; callbacks map onto the existing `DexStatus` values so themes/overlay/notch/session relay work unchanged), `pcm-player.ts` (gapless scheduled AudioBuffers through an AnalyserNode → `getAmplitude()` shows the model's real voice), `pcm-capture.ts` (AudioWorklet on the session mic stream; one shared 24kHz AudioContext both directions). In `use-dex.ts`: the **wake word still gates the session** — all wake paths set `realtimePendingRef` and funnel through `startMode("command")`, whose realtime branch calls `startRealtimeConversation` instead of STT capture. Barge-in = server VAD (`speech-started` flushes local playback); the wake-word barge monitor is never armed. No session resume: reconnects seed the last ~10 turns via a context item. 25-min gateway cap / 5-min idle kill degrade to `onDisconnect` → back to wake. **Single-session invariants** (two live sessions = two voices talking over each other): the renderer guards the multi-await connect flow with a generation token (`realtimeGenRef` — every teardown/newer connect bumps it; a superseded connect ends its half-started session at its next checkpoint), and main keeps one session per sender (`realtimeSessionBySender` — a new `realtime:start` ends the sender's previous session).
- **Hybrid tools**: enabled skills without `SkillMeta.imageResults` are declared directly to the session (`src/main/agent/realtime/realtime-tools.ts`, zod → `z.toJSONSchema`) and execute in main next to the socket, permission-gated per session via `buildToolSet({include})`. Image-returning skills (computer) are reachable only through **`run_task` delegation**: the model calls `run_task(task)` → notice to the renderer → `runDelegatedTask` drives the EXISTING `window.opendex.chat()` pipeline (configured `llm` model, full toolset, permission popup, activity banners) with an isolated message array; sub-agent sentences/actions are injected as `[task progress]`/`[task action]` context items and a throttled `response-create` makes the realtime voice narrate; the final text answers the tool call. Instructions: `buildRealtimeInstructions` (persona + direct-skill manuals + realtime addendum; briefing folded in on greeting sessions).
- Smoke test without Electron/audio: `pnpm smoke:realtime [model-id]` (token mint, text turn, tool round-trip). Grok caveat: speech-to-speech only — no transcripts, so the UI shows no text for its sessions.

## Skills + permission gate (Phase 5a)
- **Skills are self-contained folders** under `src/skills/<name>/` (clock, weather, web-search, open, computer), each spanning the process boundary: `meta.ts` (dependency-free id/label/sensitivity + tool-name constants, shared), `skill.ts` (the executable `Skill` — zod/node/ai, MAIN only), `view.tsx` (optional renderer UI: tool labels + result cards, auto-registered). The two halves link only by the tool `name` string declared in `meta.ts`. A skill bundles tools `{ name, description, inputSchema (zod), execute }`; `sensitive: true` routes every call through the permission gate. `src/skills/registry.ts` `buildToolSet({config, requestPermission})` assembles enabled skills' tools (explicit `BUILTIN_SKILLS` list — kept explicit so `pnpm smoke:chat` works under tsx), wrapping sensitive tools' `execute` to `await requestPermission(...)` first. The renderer side (`@skills` alias): `metas.ts` + `tool-views.ts` auto-discover `meta.ts`/`view.tsx` via `import.meta.glob`. **To add a skill, drop a folder — see CONTRIBUTING.md.**
- **Permission gate** (`agent/permissions.ts`): per-request `makePermissionRequester(sender)` checks the standing decision in `config.skills.permissions[id]` (`always`→run, `never`→deny, else prompt via the injected `PermissionUi` → dedicated popup window → `permission:respond`). `allow_once|always|deny|never`; `always`/`never` persist via `updateConfig`. The `chatStart` handler builds tools per-turn with a sender-bound requester. (The prompt UI now lives in its own always-on-top popup window, not the main window — see "Window model".)
- Config: `skills: { enabled, permissions }`. A skill may set `optIn: true` (off until the user enables it — see computer-use); `isSkillEnabled(skill, config)` is the single source of truth (`optIn` → on only when `enabled[id] === true`, otherwise on unless `=== false`). Renderer: `components/permission-prompt.tsx` overlay + `lib/use-permission.ts` (queues prompts), rendered as global chrome in `App.tsx`; Settings "Skills & tools" section (enable toggle + Ask/Always/Never) driven by `@skills/metas` (`SKILL_METAS`, auto-discovered from each skill's `meta.ts` — dependency-free, so the renderer never pulls the electron-backed `skill.ts`).
- **Session grant:** `makePermissionRequester(sender)` holds a per-requester `sessionAllow` set, so an "Allow once" approval covers the rest of *that command's* multi-step tool loop (a fresh requester is built per `chatStart`, so it never silently persists across commands). Needed so computer-use doesn't re-prompt on every click.

## Computer-use (Phase 6) — see & control the desktop
- **Model-agnostic, gated.** `src/skills/computer/` is the `computer` skill (`sensitive`, `optIn`; `skill.ts` + `screen-capture.ts` + `view.tsx`). Tools: `captureScreen` (optional `region` to **zoom** + `displayId` to pick a monitor), `click`, `moveMouse`, `drag`, `typeText`, `pressKeys`, `scroll` (optional `x,y` to aim the target pane), `wait`. Acting tools return a **fresh screenshot to the model** via the AI SDK's `toModelOutput` content path (`{type:"content", value:[{type:"text"},{type:"media", data, mediaType}]}`) — so the screenshot→act→screenshot loop works through the AI Gateway with **any vision model**, not just Anthropic's beta computer-use tool. `SkillTool.toModelOutput` is threaded through `buildToolSet` → `tool()`.
- **Adaptive screenshot cadence** (`finishAction`): not every action snapshots. `click`/`drag`/`scroll`/`wait` default to returning a screenshot (they change the view); `typeText`/`pressKeys` default to **none** so the model can chain related keystrokes without a round-trip per key. Every acting tool takes an optional `screenshot` boolean to override, and the system prompt instructs the model to batch and only look when it needs to. Pairs with the `prepareStep` pruning (keep last 2 images) to keep the loop fast.
- **Settled + diffed frames.** When an action does snapshot, it goes through `captureStable()` (re-captures until two consecutive frames stop differing, capped ~1s) so the model never acts on a half-loaded/spinner frame, then frame-diffs against the last frame the model saw (32×32 grayscale `signature` + `framesDiffer`): if nothing changed it returns a short **"no visible change"** text note and **omits the image** (cheaper, and a misclick signal). Net per-step image payload trends down.
- **Human-like mechanics.** Long `typeText` is **pasted** via the Electron `clipboard` (save → write → ⌘/Ctrl+V → restore) instead of per-keystroke; short text still types. `drag` uses nut.js press/animated-move/release. Cursor moves **animate** (`mouse.move`+`straightTo`) so the session is watchable; the `computer.animateCursor` config flag (Settings → Skills) flips to instant `setPosition` for max speed.
- **Screen capture** (`src/skills/computer/screen-capture.ts`): Electron `desktopCapturer` + `screen` (no native module), captured at **native** resolution (`logical × scaleFactor`, crisp on Retina) then downscaled to ≤1280px wide; an optional `region` is cropped **before** downscale so a small area renders at full detail within the budget. Defaults to the display under the cursor. The `Screenshot` carries `{offsetX,offsetY,scaleX,scaleY}` and `toScreenPoint` maps screenshot-pixel coords back to global logical display coords (handles multi-monitor + zoom crops).
- **Input control:** `@nut-tree-fork/nut-js` (maintained nut.js fork) — the project's first **native dependency**. Its `libnut` binding is N-API (ABI-stable across Electron), ships prebuilt per-platform binaries, and is externalized by electron-vite / shipped by electron-builder. This is a deliberate departure from the "pure JS/WASM" voice stack: OS-level input injection has no portable JS path.
- **System prompt:** computer-use is **not special-cased** — the skill declares its operating manual via the generic `Skill.systemPrompt` (`src/skills/computer/skill.ts`: screenshot-first, platform-correct modifier keys, small steps, permission hints). `buildSystemPrompt({config, briefing, skillPrompts})` appends every enabled skill's `systemPrompt` on non-briefing turns; `chatStart` gathers them via `skillSystemPrompts(config)`. The tool-loop budget is a flat `MAX_STEPS = 40` for all turns (`chat.ts`), no per-skill bump.
- **TTS:** narration streams to TTS sentence-by-sentence for every turn, computer-use included (the old computer-only "quiet mode" pacing was removed). The prompt asks the model to keep narration brief during screen control; everything still shows in the transcript.
- **Interrupt / emergency stop:** `useDex().interrupt()` aborts the in-flight command (model stream + tool loop) and returns to passive wake without tearing down the session — distinct from `stop()` (full teardown) and from Mute (which only silences TTS). It's driven by a **global** hotkey `CommandOrControl+Escape` (registered in main like push-to-talk → `interrupt` IPC → preload `onInterrupt` → hook), so it works even while the controlled app has focus. `App.tsx` also renders a visible `StopControl` (top-center) whenever status is `thinking`/`speaking`. Typing a new command (`submitText`) also interrupts.
- **OS permissions:** macOS needs **Screen Recording** (else captures come back empty — surfaced as a spoken hint) and **Accessibility** (for input; silently no-ops if missing). Grant them to the Electron/packaged app in System Settings → Privacy & Security.

## Window model + always-on visibility (Phase 7)
- **One persistent main window hosts the whole voice session** (mic/STT/TTS/`useDex` run only in its renderer). It is **hidden, never destroyed**: `close` is intercepted → `hide()` (real quit only via tray/⌘Q, gated by `isQuitting`), and `webPreferences.backgroundThrottling:false` keeps the wake/STT/meter timers full-speed while hidden. So the session survives out of sight; the overlay + notch are how you see/drive it.
- **Three renderer experiences, one bundle**, selected by URL hash in `main.tsx`: `#settings` (settings window), `#overlay` (the action HUD), default (main voice experience). `document.documentElement.dataset.window` reflects which.
- **Session-state relay:** the live snapshot of what the assistant is doing (`SessionState`: status/muted/bargeIn/activity/captions) lives only in the main window's `useDex`. A `useEffect` publishes it via `window.opendex.publishSessionState` → `session:update` IPC → main caches `latestSessionState` + re-broadcasts `session:changed` to view-only surfaces. New windows get the cached snapshot on `did-finish-load`.
- **Overlay HUD** (`OverlayApp`, `createOverlayWindow`): a transparent, click-through (`setIgnoreMouseEvents(true,{forward:true})`), always-on-top (`screen-saver` level + `setVisibleOnAllWorkspaces({visibleOnFullScreen:true})`) window that floats the `ToolActivityBanner` + `StopControl` over the whole desktop — visible even over another app's fullscreen space (the computer-use case). It shows only while busy (`showInactive`, never steals focus) and hides at rest, both driven by `broadcastSessionState`. Positioned a notch-height below the top edge (`positionOverlay`) so its banners never collide with the notch bar. The Stop button arms pointer events on hover (`setOverlayInteractive` → toggles `ignoreMouseEvents`) and relays to the main window's interrupt path (`overlay:interrupt`). **These hints/Stop are intentionally NOT rendered in-window anymore — the overlay owns them.**
- **Notch / compact mode** is **automatic, not a setting** (`WindowMode` is runtime-only state in main, never persisted), and the notch is its **own window** — not the reshaped main window. This is deliberate: the main window is opaque and macOS rounds its corners with no runtime API to square them, so a reshaped main window always looks like a floating rounded pill, never "part of the notch." `createNotchWindow` makes a **transparent, frameless, always-on-top** window so CSS draws the shape: `CompactBar` is `rounded-b-2xl` only → flat top edge **flush to the screen** (`placeNotch` uses `display.bounds`, y=0, `screen-saver` level over the menu bar), rounded bottom — it hangs from the top. The bar is **content-sized**: a `ResizeObserver` measures the rendered box and drives the window via `setNotchSize` (IPC `notch:set-size`), kept centered — so it's compact at rest (status indicator + mic, **no idle text**) and grows, animated, for a caption / result card / type field. The **status indicator is themeable**: the active theme's `DexThemeDef.StatusIndicator` (resolved in `NotchApp` via `getDexTheme`, default `StatusDot`) renders in the bar — e.g. editorial's LED dot-matrix. `applyWindowMode` **swaps windows** (notch → `notchWindow.showInactive()` + hide main; full → hide notch + show/focus main); the session always lives in the (now hidden) main window. `attachAutoModeListeners` engages notch on main **blur** (clicked away / agent drives another app — both lose focus; skipped while a permission prompt holds focus); returning to full is explicit (the notch's expand button). The notch is a **view**: it reads the session snapshot (`onSessionState`) and relays actions back to the session via `view:command` → main → `remote:command` → the main window's `useDex` (`submitText`/`toggleMute`; `expand` switches mode). Gated on `onboarding.completed` so the wizard always runs full.
- **Permission prompts live in their own popup window** (`createPermissionWindow` → `#permission` → `PermissionApp`): a dedicated, transparent, always-on-top (`screen-saver` + `visibleOnFullScreen`), focusable window, so a sensitive-tool prompt is visible whatever the main window is doing (hidden / notch / behind the driven app) **and answering it never disturbs the main window's layout**. `permissions.ts` is decoupled from windows via an injected `PermissionUi` (`setPermissionUi({present,dismiss})`, wired in `index.ts`): `present` centers+shows+focuses the popup and sends `permission:request` to it (deferred to `did-finish-load` on first open); `dismiss` (called from `settle`, i.e. every resolution path — answer/timeout/window-death) sends `permission:dismiss` so the popup drops a stale prompt, and hides the popup once `pendingPermissions()` hits 0. `use-permission` queues requests and prunes on dismiss. The popup is created hidden at startup and reused.
- **Summon hotkey** (`hotkeys.summon`, default `Alt+Space` = ⌥Space; `registerSummonHotkey` falls back to `Control+Alt+Space`/`Control+Shift+Space` if it can't register, e.g. Windows reserves Alt+Space): `summonWindow` toggles whichever surface the current mode uses — the notch window in notch mode (Spotlight-style focused bar to type into), the main window in full — and emits `window:summoned` → that window's renderer (`App.tsx` or `NotchApp`) dispatches a `opendex:summon` event → its `TextComposer` reveals + focuses. Editable via a key-capture field in Settings → Appearance; re-bound live on `config:set`.
- **Tray** anchors everything when all windows are hidden: Show / Settings / Quit. `app.on("activate")` (dock click) re-summons rather than spawning a duplicate.
- **Cross-platform caveats:** transparency + over-fullscreen are reliable on macOS/Windows, WM-dependent on Linux (degrades to a normal top-most strip). Tool-activity banners + Stop relocated from in-window to the overlay window.

## Tool-activity banners
- The agent loop streams **tool-call events** to the renderer, not just text: `streamChat` iterates `result.fullStream` and fires `onToolCall({toolCallId,toolName,input})` on each `tool-call` part → `chatStart` forwards them over the `chat:tool:<id>` IPC channel (`ToolCallEvent`) → preload's `chat()` surfaces them via `onToolCall` → `useDex` turns each into a transient `ToolActivity` (auto-expires after 4s) via `lib/format-tool-call.ts` (toolName+input → icon+label). These flow into the published `SessionState.activity` (gated on `appearance.showToolActivity`) and render in the **overlay HUD window** (`ToolActivityBanner`, top-center) — see "Window model" above; they are no longer rendered inside the main window.
- The chat loop also prunes stale screenshots between steps: `prepareStep` keeps only the last `KEEP_SCREENSHOTS` (2) image tool-results and stubs older ones out — keeps computer-use latency flat instead of growing per step. Screenshots are JPEG (≤1280px) for smaller per-step uploads.
